# Render thread-scaling baseline — 2026-07-10

Generated by `benchmarks/thread_scaling.py` (JIT enabled). Each thread count runs
in its own subprocess with `NUMBA_NUM_THREADS` set; the parent aggregates warm-ms,
speedup vs 1 thread, tracemalloc peak, and a cross-thread output-hash check.

Host: 8 logical CPUs. Command:

```
.venv/Scripts/python.exe -m benchmarks.thread_scaling --tier quick --repeats 3
```

Stall stays ~22 ms at every thread count — the prange kernels release the GIL, so a
background render does not freeze the UI thread.

## Takeaways for Task 4.4 (bounded threading policy)

- **Diffusion stays sequential.** It shows ~1× across all counts (per-pixel serial
  error propagation); never hand it more than one thread.
- **Parallel color kernels give diminishing returns past 4 threads.** `ordered`
  scales best (1.99× → 2.40× → 2.57×); `nearest` plateaus at ~1.4× by 2 threads;
  `ramp` peaks ~1.3× at 4. A ~4-thread cap captures most of the win without
  oversubscribing an 8-CPU host.
- **Effects regress with more threads** (0.78× at 8t): the heavy stack is
  PIL/NumPy/GIL-bound, and extra Numba threads only add scheduling overhead. Keep
  GIL-bound effect work at 1–2 threads.
- **Interactive responsiveness is already safe** off the GUI thread; the policy's
  job is throughput bounding, not heartbeat rescue.

Suggested Auto default for 4.4: clamp interactive Numba threads to
`min(4, cpu_count)`; reserve UI capacity during exports; never parallelize
diffusion.
